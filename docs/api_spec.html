シーケンス図: 原稿一覧・詳細取得

原稿一覧画面の表示から、特定原稿の詳細取得までのフロー

やっていることの流れ（かんたん解説）

1原稿一覧を表示する
  • ユーザ
    ブラウザで原稿一覧画面を開きます。
  • 画面（Frontend）
    裏側のサーバーに「原稿の一覧をちょうだい」とリクエストを送ります。
    API呼び出し: GET /source-images
  • サーバー（Backend API）
    データベースから原稿の情報（タイトル・サムネイルのキーなど）をまとめて取得します。
  • サーバー → ファイルストレージ
    サムネイル画像のキーをもとに、ブラウザで表示できる画像URLを生成します。
    画像ファイル自体はクラウドストレージに保管されており、一時的な表示用URLを発行する仕組みです
  • 画面（Frontend）
    受け取った情報をもとに、サムネイル画像の一覧をユーザに表示します。
2原稿の詳細を見る
  • ユーザ
    一覧の中から見たい原稿をクリックして選択します。
  • 画面（Frontend）
    サーバーに「この原稿の詳しい情報をちょうだい」とリクエストを送ります。
    API呼び出し: GET /source-images/{sourceImageId}
  • サーバー（Backend API）
    データベースからその原稿の詳細情報（タイトル・説明・画像キーなど）を取得し、ファイルストレージからフルサイズの画像URLを生成します。
  • 画面（Frontend）
    原稿の詳細情報と大きな原稿画像をユーザに表示します。

シーケンス図: AI画像生成

特定原稿からテキスト差し替え画像を生成し、結果を取得するまでのフロー

やっていることの流れ（かんたん解説）

1元になる原稿を開く
  • ユーザ
    テキストを差し替えたい元原稿の詳細画面を開きます。
  • サーバー（Backend API）
    「原稿一覧・詳細取得」と同じ流れで、DBから原稿情報を取得し、ストレージから画像URLを発行して画面に返します。
  • 画面（Frontend）
    原稿の詳細と画像を表示します。ここからユーザはテキスト差し替えの操作に進めます。
2AIに画像生成を依頼する
  • ユーザ
    元のテキスト（originalText）と差し替え後のテキスト（replaceText）を入力して「生成」ボタンを押します。
  • 画面（Frontend）
    サーバーに「この原稿のテキストを差し替えた画像を作って」とリクエストを送ります。
    API呼び出し: POST /generated-images { sourceImageId, originalText, replaceText }
  • サーバー（Backend API）
    すぐに「受け付けました（生成中）」と返事を返します。画面にはローディング表示が出ます。
    画像生成には時間がかかるため、結果を待たずに先に「受付OK」を返す仕組み（非同期処理）です
3裏側でAIが画像を生成する（バックグラウンド）
  • サーバー（Backend API）
    元原稿が本当に存在するかDBで確認し、生成画像DBに「処理中」のレコードを作成します。
  • サーバー → 生成AI
    元原稿画像と差し替えテキストを生成AIに渡して、テキストが差し替わった新しい画像を作ってもらいます。
  • サーバー → ファイルストレージ
    AIから返ってきた生成画像をストレージに保存します。
  • サーバー（Backend API）
    生成画像DBのレコードを「完了」に更新し、画面に「生成が終わりました」と通知します。
4生成結果を確認する
  • ユーザ
    生成完了の通知を受けて、生成結果を表示する操作をします。
  • 画面（Frontend）
    サーバーに「この生成画像の情報をちょうだい」とリクエストを送ります。
    API呼び出し: GET /generated-images/{generatedImageId}
  • サーバー（Backend API）
    生成画像DBから結果を取得し、ストレージから表示用の画像URLを発行して画面に返します。
  • 画面（Frontend）
    AIが生成したテキスト差し替え済みの画像をユーザに表示します。

確認事項

04/17 共有のAPI仕様を確認した上で、ブラザー様に確認したい事項（04/20 作成）

A データの保存先・アップロード 最優先
A-1. 元原稿データのアップロード手段が未定義
API仕様上、Source Image DB（DB1）はFrontendから Read Only で、取得系APIしか存在しません。
GET /source-images（一覧）と GET /source-images/{sourceImageId}（詳細）のみ。

一方、Bubble側には「デザインテンプレート・共通印刷物の原稿デザインデータをアップロードする画面」が存在します。（宮田さん 04/17 メールで言及）
  • 元原稿データの登録はブラザー側が管理者として直接DBに投入する想定か？
  • それとも Bubble側からアップロードするAPIが今後追加される予定か？（例: POST /source-images）
  • デフォルトの原稿データはブラザー側サーバーに保存し、BubbleからはAPIで取得するだけ、という認識で合っているか？
Bubble側のアップロード画面を残すべきか、削除してAPI経由の表示だけにすべきかの判断に直結します。
A-2. Bubble側にデータを保存する必要があるケースの有無
API仕様では画像・生成データはすべてBrother側に保存される設計です。
  • Bubble側のDBには何も保存しなくてよいか？（キャッシュ・一時保存も不要？）
  • ユーザーのお気に入り・閲覧履歴など、Bubble側で独自に管理すべきデータはあるか？
B 非同期処理の完了通知方式 最優先
B-1. 生成完了をFrontendにどう伝えるか
シーケンス図では以下の流れが描かれています:

POST /generated-images → status=RUNNING を即時返却
  ↓（バックグラウンド処理）
API → FE: generatedImageId, status=COMPLETED

この 「API → FE: status=COMPLETED」の通知方法が不明です。
  • Webhook方式: 生成完了時にBubble側のURLに通知を送ってもらう形か？
  • ポーリング方式: Bubble側が定期的に GET /generated-images/{id} でステータスを確認する形か？
  • その他の方式（WebSocket、Server-Sent Events 等）を想定しているか？
Bubble側としてはWebhook方式が望ましいです。ポーリングの場合は推奨間隔も知りたいです。Bubble側の実装パターンが根本的に変わるため、最優先で確定したい事項です。
B-2. 生成にかかる想定時間
  • 1回の画像生成の想定所要時間は？（数秒 / 数十秒 / 数分）

F 実際の画面フローとAPIのギャップ（04/20 追記） 最優先
F-1. リクエストの情報量が全然違う
現在のAPIの生成リクエスト:
POST /generated-images { sourceImageId, originalText, replaceText }

実際に送りたい情報（画面から読み取れるもの）:
行事名、開催期間、行事詳細、プログラム（複数行）、注釈、連絡先 等

originalText / replaceText の1組だけでは、複数のテキスト項目を背景画像に配置できない。
  • リクエスト構造を拡張して、複数のテキストフィールドを渡せるようにするか？
  • それとも originalText/replaceText を配列にして複数組渡す設計か？
F-2. テンプレートの階層構造がAPIにない
実際のデータ構造（Bubble側）:

デザインテンプレート（管理者が作る）
  └ 1デザイン × 7サイズの背景画像

行事テンプレート（管理者が作る）
  └ デザインテンプレートを最大3種紐づけ
  └ 業界・月で分類

行事（ユーザーが作る）
  └ 行事テンプレートを選択（or なし）
  └ 文字情報を入力
  └ 生成ボタンで 3デザイン × サイズごとに生成

今のAPIの構造:
Source Image = フラットな一覧（階層なし、グルーピングなし）
Generated Image = 1回のPOSTで1枚だけ生成
  • テンプレートの親子関係（デザインテンプレ→行事テンプレ→行事）はBubble側で管理し、APIにはsourceImageId単位で投げる想定か？
  • それともAPI側にもテンプレート階層を持たせるか？
F-3. 一括生成（3デザイン × 複数サイズ）
ユーザーが「生成」ボタンを1回押すと、最大 3デザイン × 4サイズ = 12枚 の画像生成が発生する。
現在のAPIは1回のPOSTで1枚だけ生成する設計。
  • 12回個別にPOSTする想定か？一括生成APIを用意するか？
  • 個別POSTの場合、12件の非同期完了通知をどう管理するか？
F-4. 背景画像の保存先
管理者がデザインテンプレートで 7サイズの背景画像をアップロード している。
APIには POST /source-images がない。
  • 背景画像はBubble側に保持して、生成時にURLでAPIに渡す想定か？
  • それともBrother側にアップロードAPIを追加するか？
F-5. サイズ指定の方法
画面では A3 / A4 / 長尺縦 / 長尺横 のタブがある。
APIにはサイズを指定するパラメータがない。
  • サイズごとに別の sourceImageId として管理する想定か？
  • それともリクエストパラメータでサイズを指定するか？
