## 1. 結論

- **429は解消を確認**。6サイズほぼ同時投入（バースト）を2ラウンド連続（計12ジョブを約2分間に投入）実施し、**全12件がCOMPLETED、429発生ゼロ**
- さらに**5ユーザー同時利用想定（5グループ×6サイズ＝30ジョブを2秒以内に同時投入）でも全30件COMPLETED、429発生ゼロ**（7章）。全件完了まで約3分36秒（スループット実測 約8〜9件/分）
- 前回（7/8）は10秒間隔・完全直列でも6件中3件が429だったのに対し、今回はそれより厳しい同時投入でも全件成功。**毎分1件程度と推定していたクォータは実質的に撤廃されたと判断できる**
- Bubble側の429限定自動リトライ（検証まとめ2-3）は保険として実装しておく価値はあるが、**dev環境での画面検証のブロッカーは解消**
- あわせて**API Key認証が有効化されていることを確認**（下記4。Bubble側API Connectorへの `x-api-key` 設定が必須）

## 4. API Key認証の有効化を確認（構成変更）

7/6検証時はAPI Keyなしで200だったが、今回は以下のとおり**API Key必須に変更されていた**。

- `x-api-key` なし: **403**
- `x-api-key` あり（連携ガイド記載のキー）: **200**

「Bubble側担当者向け API Key設定 / WAF IP WhiteList 連携ガイド」2章のとおり、**Bubble側API Connectorの全対象リクエストに `x-api-key` ヘッダーの設定が必要**（未設定だと生成以前に403で失敗する）。

## 5. 今回気づいた事項（429とは別件）

### 5-1. 長尺横の生成サイズが不安定（新規・要確認）

同一ソース（src-c4144f6c）・同一ペイロードで、ラウンド1とラウンド2の出力実寸が異なった。

| ジョブ | 出力実寸 | 判定 |
|---|---|---|
| ラウンド1 gen-745b535d | **1376×768px（約16:9）** | NG（長尺比率でない） |
| ラウンド2 gen-2f7b81fa | 1793×592px（約3:1） | OK（7/6検証時と同等） |

ソース画像（サムネイル3402×1123px、約3:1）に対し、ラウンド1のみアスペクト比が大きく異なる出力となった。再現性・原因（AI側の出力サイズゆらぎか）はブラザー側への確認候補。

### 5-2. 未送信項目の位置にテンプレ由来文言が残る（既知挙動の再確認）

最小リクエスト（eventName / eventPeriod のみ送信）のため、未送信項目の位置にソーステンプレート由来の文言がそのまま描画される。

- A4: 「10:30 勤行 / 11:15 法話 / 兄弟舎主宰 堀田 太郎 師 演題:『つながる命、感謝の心』」「〇〇寺 〇〇市〇〇区〇〇町〇〇 TEL〇〇〇-〇〇〇-〇〇〇〇」等
- 長尺縦: 「〇〇寺」「令和八年 〇月〇〇日」
- ラウンド1長尺横: 「〇〇寺」

eventName / eventPeriod 自体は全サイズで正しく描画されており、7/6のissue（再生成時の文言混入）の再発ではない。実運用でBubble側から全項目を送信する場合の影響は限定的だが、ユーザーが項目を未入力にした場合の挙動として認識しておく（ブラザー側PDFのテスト結果でも同様に「〇〇」プレースホルダが残っている）。

## 7. 追加検証: 5ユーザー同時利用想定の負荷テスト（18:29〜18:33）

本番で複数ユーザーが同時利用する想定（検証まとめ1章で「成立しない」としていた条件）を模擬した。

### 条件

- 5ユーザーがそれぞれ別のソースグループ（urabon1 / urabon2 / urabon3 / jyuya1 / akihigan1）を使用
- 各ユーザーが画像生成対象の6サイズ（A4 / A3 / 長尺縦 / 長尺横 / ハガキ / 封筒(長形3号)）を一括生成
- **計30ジョブを並列POST（18:29:42〜18:29:44、2秒以内に全件投入）**
- ペイロードは最小リクエスト（eventName: 「同時利用テスト ユーザーN」、eventPeriod のみ）

### 結果

- POST: **30/30が200で受理**（`status: RUNNING`）
- 最終結果: **30/30がCOMPLETED、FAILED（429含む）ゼロ**
- 完了推移: 投入13秒後に2件完了 → 以降ほぼ一定ペース（約2件/10秒）で進行 → **18:33:18に全30件完了（投入から約3分36秒）**
- バックエンド側でジョブが整列処理されていると見られ、同時投入してもクライアントからは「処理時間が延びるだけで失敗しない」挙動。スループット実測は約8〜9件/分
- 長尺横の出力実寸も5ユーザー全件1793×592px（正常比率）で、5-1のサイズ異常は本ラウンドでは再発せず
