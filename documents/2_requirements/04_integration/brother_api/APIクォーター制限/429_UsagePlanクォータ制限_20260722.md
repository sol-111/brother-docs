- 対象環境: brother-backend-dev-yokinini
- Base URL: `https://u7ib5ltbdd.execute-api.ap-northeast-1.amazonaws.com/Prod`
- 経緯: 生成API・再生成APIのクォータ制限（AI側429）の再発有無を確認するため実測したところ、生成系に限らず**全APIが429を返す状態**を確認した。エラー種別から、7/8に観測したAI側クォータとは異なる**API Gateway Usage Plan（API Keyごとの利用回数上限）のクォータ枯渇**と判断

## 1. 結論

- **全APIが429（`LimitExceededException` / `{"message":"Limit Exceeded"}`）で利用不可**。一覧取得・生成・再生成・原稿DL URL取得のすべてがAPI Gateway入口で拒否される
- 本事象は7/8のAI側クォータ429（生成ジョブのみFAILED）の再発ではなく、**7月上旬のAPI Key認証導入時に設定されたUsage Planの期間クォータ（日次等の総利用回数上限）を使い切ったことによる新規事象**
- クォータの上限値・期間・リセットタイミングは設定者（ブラザー側）にしか確認できないため、**設定値の共有とdev検証用の上限引き上げをブラザー側へ依頼したい**（4章）
- 本来の確認目的だった生成・再生成のAI側クォータ再発有無は、本事象により**未確認**（クォータ解消後に再検証する）

## 3. 事象の解説: 7/8のAI側429との違い

API Gatewayでは、API Keyに対してUsage Plan（利用量プラン）が紐づき、以下が設定できる。

| 設定 | 意味 | 超過時のエラー種別 |
|---|---|---|
| Rate / Burst | 秒間リクエスト数の制限（スロットリング） | `TooManyRequestsException` |
| **Quota** | **期間（日/週/月）あたりの総リクエスト数上限** | **`LimitExceededException`** ← 今回 |

今回観測したのはQuota超過であり、「瞬間的に叩きすぎ」ではなく「**期間内の総利用回数を使い切った**」状態。期間が切り替わる（またはブラザー側がリセット/引き上げする）まで、全リクエストが拒否され続ける。

7/8の429との比較:

| | 7/8の429 | 今回（7/22）の429 |
|---|---|---|
| 発生階層 | AI生成側（バックエンドの奥） | **API Gateway入口** |
| 挙動 | POSTは200で受理→ジョブがFAILED（`AI_REQUEST_FAILED:429`） | **リクエスト自体が429で拒否** |
| 影響範囲 | 画像生成ジョブのみ | **全API**（一覧表示・DLも不可） |
| 状態 | 7/16に解消確認済み | **本件（未解消）** |

```
[Bubble] → [API Gateway (Usage Plan)] → [ブラザーバックエンド] → [AI]
                 ↑今回の429はここ              ↑7/8の429はここ
```

## 4. ブラザー側への依頼事項

1. **Usage Plan設定値の共有**
   - Rate / Burst / Quotaの各上限値、Quotaの期間（日/週/月）とリセットタイミング
2. **dev検証用のQuota上限引き上げ（または一時リセット）**
   - 開発・検証中は生成状態のポーリング等でリクエスト回数を多く消費するため、本番想定より緩い設定が必要
3. **検証再開可能時期の連絡**
   - 現在全APIが利用不可のため、Bubble側の画面検証・アイビス側のAPI検証がともにブロックされている

## 5. 本番運用に向けた論点（メモ）

- Quota設定自体はAPI Key漏洩時の被害限定として妥当なセキュリティ運用であり、撤廃ではなく**適正値へのチューニング**が必要
- 本番の適正値は「想定同時利用ユーザー数 × 1ユーザーあたりの操作回数（一覧取得＋生成6サイズ＋ポーリング回数＋DL）」から見積もる。特にポーリング（10秒間隔×生成所要10〜60秒×6サイズ）が支配的になるため、ポーリング間隔の設計とセットで検討する
- クォータ超過時にBubble側がどう縮退するか（表示のみ利用可にする等）も本番前に設計しておく
