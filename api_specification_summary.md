# API仕様まとめ（議事録ベース）

> 議事録 2026/2/18 ~ 2026/4/9 の内容から抽出・整理したもの。
> API仕様は4月第2週を目途に最終確定予定（担当: 柳）。

---

## 1. 全体アーキテクチャ

### システム構成
- **フロントエンド**: Bubble（バブル）で構築
- **画像生成**: ブラザー側 AI チームが提供するAPI
- **連携方式**: Bubble → トリガーAPI → 画像生成API（トリガーAPIを間に挟む構成）
- **設計方針**: 汎用性・保守性を重視し、将来的な拡張や他サービス展開も見据えた設計

### トリガーAPI方式（3/18決定）
- API呼び出し時に**全情報を渡し**、トリガーAPI側で必要項目を選択・処理する
- 将来的な拡張性・保守性を重視する方向

---

## 2. 画像生成API

### 基本仕様
| 項目 | 内容 |
|------|------|
| 呼び出し単位 | **サイズごとに個別呼び出し**（2/18決定） |
| 対応サイズ | A3縦、A4縦、長尺縦、長尺横（4種。今後増加も許容） |
| デザインパターン | 3種（A/B/C）を一括または個別で生成（3/4議論） |
| サイズ・向き | API呼び出し時点で決定。ユーザーによる変更不可（3/4決定） |
| 画像生成方式 | AI側でプロンプトベースで処理（3/18確認） |
| 生成済みデータ | サーバー保存（2/18決定） |

### リクエスト（想定）
- 行事情報（行事名、開始日、終了日、詳細、注釈、プログラム等）
- テンプレート情報（使用テンプレートID、デザインパターン）
- 背景画像データ（Bubble側から渡す）
- 印刷物サイズ・向き指定
- ユーザー入力テキスト（文字数・行数制限あり）
- 連絡先情報
- 業界・サブ業種情報

### レスポンス
- **画像データはURL形式で返却**（3/25決定）
  - 理由: BubbleのAPIコネクター制限が5MBのため、直接バイナリ返却は不可
  - ダウンロードURL発行方式

### 非同期処理・タイムアウト対策（3/25, 4/1議論）
- 画像生成は時間がかかるため、非同期処理が必要
- **Webhook方式** または **ステータス取得方式（ポーリング）** を検討中
- 進捗情報の取得方法についても議論中

## 3. 表示項目の制御

### API側での項目コントロール（3/11決定）
- A3/A4等の**サイズごとに表示項目が異なる**場合、API側でコントロール
- 設定ファイル（JSON）による項目管理はAPI側で保持・参照
- 将来的な拡張にも対応可能な設計

### テキスト入力制限（3/11決定）
- 注釈やプログラム等は「**最大行数**」と「**1行あたりの最大文字数**」で制限
- 半角・全角の幅差は最大幅基準で運用
- 入力制限の注意文や簡易プレビューでユーザーをガイド

---

## 4. 画像データの管理

### テンプレート画像（3/18, 3/25決定）
- テンプレート画像は**Bubble側で管理**
- API呼び出し時に該当画像データを渡す運用
- デザインパターン（A/B/C） x サイズごとに画像を管理

### 背景テンプレートと行事テンプレートの分離（3/25決定）
- 管理コスト・一括変更性を重視し、背景と行事テンプレートを分離管理
- 画像セットの一括管理・再利用性向上

### 生成画像の保存（3/4, 3/25議論）
- 画像は一時的にAIチームサーバーに保存
- ダウンロード後は削除（長期保存・資産化は今後検討）
- 恒久保存は困難だが、POC期間中の一時保存は要相談
- **Bubble依存を避ける方向**で調整

---

## 5. デザイン生成・評価フロー

### 生成フロー（2/27決定）
1. ユーザーが行事情報を入力・保存
2. ユーザーが明示的に「**再生成**」ボタンを押す（自動生成ではない）
3. API呼び出し → デザイン3パターン生成
4. 生成画像URLがレスポンスで返却
5. ユーザーがダウンロード

- 「保存」と「再生成」を分離 → APIコスト削減とUX両立

### 評価機能（2/25, 4/1決定）
- デザイン評価は五段階評価
- POCではサイズごとに**全デザイン強制評価**
- ダウンロード時にポップアップで評価を促す
- 三案同時評価も可能
- 評価はユーザーが変更可能、変更ごとにログ記録（4/9決定）

### ダウンロード制限（2/27決定）
- ダウンロード可能な行事は「**今月 + 2ヶ月先**」まで（期間は可変対応）
- イベント終了後は該当デザインのダウンロード・閲覧不可
- 複数デザインのダウンロードは可能

## 6. ログ関連API/データ

### 操作ログ（2/25, 2/27決定）
- **CSV出力形式**（万能なCSV、分析はExcel等で実施）
- 管理画面は不要
- 期間・ユーザー・業種等で**絞り込み可能**
- 月単位でCSVダウンロード

### ログ項目
- 行事名
- ユーザー
- 使用テンプレート
- デザイン種別
- 生成回数
- ダウンロード回数・日時
- 修正回数
- デザイン評価（五段階）

### ダウンロード・評価ログ（4/9決定）
- 「**行事の月**」で仕分け

## 7. 認証・アカウント関連

| 項目 | 仕様 |
|------|------|
| アカウント単位 | 企業単位で1つ |
| 認証方式 | メールアドレス + パスワード |
| メール認証 | 導入（認証後にサービス利用開始） |
| パスワード要件 | 英数8文字以上（記号必須は不要） |
| メールアドレス変更 | 可能 |
| アカウント削除 | POC期間中は未実装（データ保持・分析に活用） |
| 複数事業所 | POCでは1アカウント1事業（将来拡張を見据えたデータ構造） |

## 8. データ構造（関連エンティティ）

### 業界
- 最大20程度
- 2階層管理（例: お寺 → 宗派）
- 追加は開発者管理

### 行事
- 開始日・終了日で管理、期間指定可能
- 同名行事の重複を許容
- ユーザーが自由に作成・削除可能
- 行事の複製機能あり

### テンプレート
- テキスト + デザインを紐付けて管理
- 業種・サブ業種ごとに用意
- 初期登録テンプレートと選択式テンプレートを区別
- デザイン数は柔軟に設定可能

### 共通印刷物
- 3パターンで分類:
  1. 全業界共通
  2. 業界別共通
  3. 第2階層共通（例: 宗派・業種）
